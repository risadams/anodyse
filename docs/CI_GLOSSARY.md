# CI/CD Glossary

Terminology and definitions for CI/CD concepts and platform-specific terms used throughout Anodyse documentation.

## Core CI/CD Concepts

### CI/CD Pipeline
**Continuous Integration / Continuous Deployment** - Automated process that builds, tests, and deploys code changes. In the context of Anodyse, a pipeline generates documentation from playbooks automatically.

**Example**: When you push code to GitHub, the CI pipeline runs Anodyse to generate documentation without manual intervention.

### Workflow
A series of automated steps triggered by events. In GitHub Actions, called a "workflow". In GitLab, called a "pipeline". The sequence of jobs/stages that execute.

**Example**: 
- GitHub: Files in `.github/workflows/` define workflows
- GitLab: `.gitlab-ci.yml` defines pipeline workflows

### Job / Stage
- **Job**: Unit of work in GitHub/CircleCI (runs independently, can run in parallel)
- **Stage**: Unit of work in GitLab/Jenkins (runs sequentially by default)

Both represent a logical grouping of steps/tasks.

**Example**:
```yaml
# GitHub Actions (Jobs)
jobs:
  test:      # This is a job
    runs-on: ubuntu-latest
    steps:   # Contains steps

# GitLab CI (Stages)
stages:
  - build    # This is a stage
  - test
  - deploy
```

### Step / Task
Individual command or action within a job/stage. The smallest unit of work.

**Example**:
```yaml
# GitHub Actions (Steps)
- name: Install dependencies
  run: pip install anodyse

# GitLab CI (Steps called "script")
script:
  - pip install anodyse
```

### Trigger / Event
Condition that starts a workflow/pipeline execution.

**Common triggers**:
- `push` - Code pushed to repository
- `pull_request` / `merge_request` - PR/MR opened or updated
- `schedule` - Cron expression (e.g., daily, weekly)
- `manual` / `workflow_dispatch` - Click button in UI
- `webhook` - External service notification

### Artifact
Output file(s) generated by a workflow and stored for later retrieval.

**Example**: Generated markdown documentation files stored for 30 days.

```yaml
# GitHub: upload-artifact action
- uses: actions/upload-artifact@v3
  with:
    path: ./docs
    retention-days: 30
```

### Secret / Credential
Sensitive information (passwords, API keys, tokens) stored securely in CI/CD platform.

**Usage**: `${{ secrets.GITHUB_TOKEN }}` or `$CI_COMMIT_SHA` (GitLab)

### Environment Variable
Named value available to all steps in a job, passed to scripts and executables.

**Anodyse variables**:
- `ANODYSE_INPUT_DIR` - Playbook directory
- `ANODYSE_OUTPUT_DIR` - Documentation output directory
- `ANODYSE_TEMPLATE_DIR` - Optional templates directory
- `ANODYSE_VERBOSE` - Enable verbose output

### Rate Limit
Maximum number of API calls or pipeline minutes per time period (usually per month).

**Example**: GitHub Actions free tier has unlimited runs but limited to 180 minutes/month for private repos.

### Concurrency
Multiple jobs/stages running simultaneously from the same pipeline or across pipelines.

```yaml
# GitHub: Prevent simultaneous runs of same workflow
concurrency:
  group: docs
  cancel-in-progress: true
```

## Platform-Specific Terminology

### GitHub Actions

**Action**: Reusable task for GitHub Actions (like `actions/setup-python@v4`).

**Runner**: Machine that executes the workflow (GitHub-hosted or self-hosted).

**Workflow**: Complete automation defined in `.github/workflows/*.yml`.

**Input**: Parameter passed to an action or workflow.

**Output**: Value returned from an action/job to subsequent jobs.

### GitLab CI/CD

**Runner**: Process that picks up and executes jobs (GitLab-hosted or self-hosted).

**Executor**: What runs the job (Docker container, shell, Kubernetes pod).

**Cache**: Store files between jobs to speed up execution.

**Artifact**: Files generated by job, passed to dependent jobs or available for download.

**Pipeline**: Complete execution of all stages defined in `.gitlab-ci.yml`.

**Rule**: Condition determining if job should run (replaces older `only`/`except`).

### Jenkins

**Pipeline**: Entire CI/CD process, usually Groovy-based (Declarative or Scripted).

**Stage**: Logical grouping of steps in Jenkins Declarative Pipeline.

**Step**: Individual action in a stage.

**Agent**: Machine/pod that executes the pipeline (or stages).

**Post**: Actions to run after pipeline succeeds, fails, or always (cleanup, notifications).

### CircleCI

**Orb**: Reusable package of configuration (like GitHub Actions).

**Job**: Unit of work that runs on an executor.

**Workflow**: Orchestrates jobs and their dependencies.

**Executor**: Specifies environment (Docker image, macOS, Windows).

**Context**: Set of environment variables available to jobs.

### Woodpecker CI

**Pipeline**: Defined in `.woodpecker.yml`.

**Step**: Individual command or action (runs in Docker container by default).

**Trigger**: Event that starts pipeline (push, PR, tag, cron).

**Matrix**: Run same pipeline with different variables (like parallel matrix).

**Plug-in**: Docker container that performs a specific task.

## Anodyse-Specific Terms

### Playbook
Ansible YAML file that defines automation tasks. Anodyse generates documentation from these.

**Anodyse processes**: 
- Playbook-level annotations (`@title`, `@description`)
- Task comments and TODO items
- Role references
- Variable definitions

### Role
Reusable set of Ansible tasks, templates, handlers, and variables organized in a directory structure.

**Anodyse generates**:
- Role overview (metadata, summary)
- Tasks listing with descriptions
- Variables and defaults documentation
- Template file listings

### Handler
Ansible task triggered by notifications from other tasks (e.g., "restart service" on file change).

**Anodyse documents**: Handler name, when they trigger, handler dependencies.

### Task
Individual action in an Ansible playbook (install package, copy file, run command).

**Anodyse extracts**: Task description, tags, conditions, affected resources.

### Annotation
Comment-based metadata that Anodyse recognizes to extract documentation.

**Format**: `# @key value` (e.g., `# @title Deploy web server`)

**Supported**:
- `@title` - Human-readable title
- `@description` - Summary
- `@param` - Parameter description
- `@warning` - Important caveat
- `@example` - Usage example
- `@tag` - Category label

### TODO / FIXME
Special comments that Anodyse collects as action items in generated docs.

**Format**: `# TODO(name): description` or `# FIXME: what's wrong`

**Anodyse includes**: Extracted TODOs in documentation under "Open Items" or "Known Issues"

### Template Override
Custom Jinja2 template used instead of Anodyse's built-in template.

**Files**: 
- `playbook.md.j2` - Template for playbook documentation
- `role.md.j2` - Template for role documentation
- `index.md.j2` - Template for index page

**Location**: `.anodyse/templates/` or custom path via `--config`

## Exit Codes

Understanding Anodyse exit codes helps with CI/CD error handling.

### Exit Code 0 - Success
Documentation generated without issues. All playbooks parsed successfully, all annotations found.

```bash
echo "✓ Documentation generated successfully"
```

### Exit Code 1 - Error
Parse error, runtime error, or missing input. Documentation generation failed.

**Causes**:
- Invalid YAML syntax in playbook
- Missing input directory or files
- Permission denied errors
- Python runtime error

```bash
echo "✗ Documentation generation failed"
exit 1
```

### Exit Code 2 - Warning
Documentation generated with warnings (non-blocking issues).

**Causes**:
- Missing `@title` annotation
- Incomplete `@description`
- Task without comments
- Referenced file not found (non-critical)

```bash
echo "⚠️ Documentation generated with warnings"
exit 2
```

## Conventions Used in Anodyse Documentation

### Code Blocks
- **YAML**: Configuration files (workflows, pipelines)
- **Bash/Shell**: Commands to run on Linux/macOS
- **PowerShell**: Commands for Windows
- **Groovy**: Jenkins Declarative Pipeline syntax
- **Markdown**: Jinja2 template examples

### Paths
- **Absolute**: `/path/to/file` (Unix-style)
- **Relative**: `./docs/`, `../` (relative to repository root)
- **Windows**: `C:\path\to\file` (when platform-specific)

### Placeholders
- `YOUR-USERNAME` - Replace with actual username
- `your-project` - Replace with actual project name
- `YOUR_TOKEN` - Replace with actual token value
- `your-branch` - Replace with actual branch name

### Links
- **Internal**: `[Link text](../path/file.md)` (relative)
- **External**: `[GitHub](https://github.com)` (absolute URLs)
- **Anchors**: `#section-heading` (within document)

### Emphasis
- **Bold**: `**important text**` - Emphasis on key concepts
- **Code**: `` `variable_name` `` - Inline code/variables
- **Blocks**: `bash`, `yaml`, `json` - Syntax highlighting
